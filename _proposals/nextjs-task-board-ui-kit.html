Three directions · grounded in actual references

Three UI-kit directions for the same task board.

Each direction below is a faithful mini-product — its own typography, surface, structural moves. No shared chrome. Palettes pulled from live inspection of real sites (Linear, Plain, Attio, Charm.sh) rather than tropes. Pick the direction that fits W5D3's voice; the kit then carries across v0 → v6.

How to read this: compare the same artifact (a status indicator, a task list row, a page composition) across the three sections. The cool/violet color in section A vs. C is incidental; the real difference is whether status is a colored dot beside slate text (A), a bracketed text flag (B), or a chevron-shape banner (C).
Direction A · Build Log

The team's narrative spine, not a kanban.

The primary surface is an activity timeline. Tasks live in the right-rail. Status is a colored shape + slate text, never a pill. Inter Variable at weights 510/590 (the variable-axis signature). Berkeley-Mono-feel monospace for IDs. Pulled directly from Linear's actual rendered structure — sherbet palette, no rounded pill backgrounds, narrow weight differentials.

CDN-042 · In progress

Wire client-side filter to server-rendered task list

Confirm the filter state lives in the client wrapper, not the server parent. The server component should still own the data read from lib/tasks.ts; the client component receives the array as a prop and slices it locally.

If the filter were a server component, every change would trigger a full server roundtrip. The whole point of "use client" in v3 is to keep interactivity in the browser without server traffic.

Wire client-side filter to server-rendered task list

Direct AI to add a TaskFilter client component that wraps the existing server-rendered task list. The server component still owns the data read from lib/tasks.ts; the client component receives the array as a prop and slices it locally.

The trap to watch for: AI will often "fix" the client component by lifting the data fetch into it. That defeats the entire point of v3. The server parent keeps the data; the client child keeps the interaction. Confirm the boundary on every diff.

How to choose between A, B, C.

Each direction works pedagogically — they all carry the same v0→v6 demo content. The pick is about voice: which surface best matches the way Marcus wants W5D3 to feel to a student arriving from the Astro event-listing demo.

If you want…To read as a real product tool — most professionally calibrated, lowest-risk, but in the genre Linear already owns.Pick A · Build Log
If you want…To read as a quiet, considered, almost-print object — most distinctive vs. event-listing, calmer tempo, requires more careful body copy.Pick B · Broadsheet
If you want…To read as a loud, playful studio piece — boldest, most opinionated, riskiest if W5D3's content is dry; rewards real personality in writing.Pick C · Inverted Studio
